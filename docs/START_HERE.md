# 🚀 התחלה מהירה - Tal Travel

**Tal Travel** - פלטפורמת תכנון וניהול טיולים אישית של משפחת טל.

## ⚡ TL;DR

אתה הולך לבנות **Tal Travel** - פלטפורמת תכנון טיולים רב-משתמשית, מודולרית, hybrid (web + mobile PWA), בסגנון TripIt עם פלטה ייחודית של לבנדר.

הטיול הראשון בפלטפורמה הוא טיול משפחתי ליפן (20.7-6.8.2026, 16 לילות). הפלטפורמה חייבת להיות מוכנה לפני הטיול, ותשמש גם לטיולים עתידיים.

| מפרט | בחירה |
|------|--------|
| **שם** | Tal Travel |
| **Stack** | Next.js 15 + Supabase + Vercel + PWA |
| **שפה** | עברית RTL ראשית, אנגלית משנית |
| **השראה** | TripIt (mobile-first UX, clean cards) |
| **צבע ראשי** | לבנדר 💜 (Lavender #8b6fb8) |
| **גישה** | Hybrid - תכנון מדסקטופ, שימוש במובייל |
| **משתמשים MVP** | Roy + Leahy + מוזמנים |

## 🗓️ תוכנית עבודה מומלצת

### שלב 1: הקמת בסיס (2-3 שעות)
- Next.js + TypeScript + Tailwind + RTL
- Supabase project (DB + Auth + Storage)
- GitHub + Vercel
- shadcn/ui + Lucide icons
- ערכת עיצוב Tal Travel (לבנדר)

### שלב 2: Schema + Auth (2-3 שעות)
- מיגרציות DB לפי `DB_SCHEMA.md`
- מסכי הרשמה והתחברות עם Magic Link
- מנגנון הזמנת משתמשים לטיול
- Row Level Security policies

### שלב 3: ניווט וטיול ראשי (2 שעות)
- Tab Bar תחתון (Trips, Today, Documents, Profile)
- מסך Trips - רשימה של כל הטיולים
- יצירת טיול חדש
- ניהול חברים והרשאות

### שלב 4: מודולים בסיסיים (6-8 שעות)
לפי סדר חשיבות לטיול יפן:
1. **Daily Plan** - מסך "היום" + ציר זמן
2. **Documents** - העלאה, קטגוריות, תצוגה (Roy ביקש במיוחד)
3. **Hotels** - כרטיסים עם סטטוס
4. **Flights** - כרטיסי טיסות מסודרים
5. **Tasks** - רשימת משימות עם התקדמות
6. **Attractions** - רשימה לבחירה
7. **Weather** - תחזית בסיסית

### שלב 5: זריעת נתוני יפן (1 שעה)
הזנת כל הנתונים מ-`JAPAN_TRIP.md` לבסיס הנתונים.

### שלב 6: PWA + Mobile polish (2-3 שעות)
- next-pwa setup
- Service worker לאוף-ליין
- אופטימיזציות מובייל
- אפשרות "הוסף למסך הבית"

### שלב 7: Deploy + הזמנה (30 דקות)
- Deploy ל-Vercel
- חיבור דומיין (אופציונלי)
- הזמנת לאהי לפלטפורמה
- בדיקה משותפת

**סה"כ זמן משוער: 15-20 שעות עבודה ב-Claude Code**

## 💡 טיפים חשובים

### העדפות של Roy

- **בלי em dashes** (--) בשום מקום
- **תאריכי לילות:** תמיד צ'ק-אין → צ'ק-אאוט. "23.7 → 26.7 · 3 לילות"
- **עברית טבעית** - לא תרגום מאנגלית
- **תמציתי אבל חם**
- **מינימום אמוג'י** - רק כשבאמת מוסיף ערך
- **תקבל החלטות בשיחה** - הוא יחליט בזמן הפיתוח

### עבודה משולבת תכנון + פיתוח

Roy ימשיך לקבל החלטות על הטיול תוך כדי הפיתוח. הזרימה:
- הוא שואל: "איזה Mimaru?"
- אתה מציע אופציות, הוא בוחר
- אתה מעדכן **גם** את `JAPAN_TRIP.md` **וגם** את ה-DB
- ככה ה-DB תמיד תואם למסמכים

### עקרונות פיתוח קריטיים

1. **Hybrid באמת** - הדסקטופ צריך להיות טוב לתכנון, המובייל מעולה לשימוש בטיול
2. **Multi-tenancy מההתחלה** - לא לכתוב הנחות שיש רק טיול אחד
3. **לא לקודד-קשה "יפן"** - הפלטפורמה גנרית, יפן זה רק נושא של הטיול
4. **מובייל = הבדיקה האמיתית** - מסך 375px לפני "מוכן"
5. **טוב מספיק מעל מושלם** - MVP יותר חשוב מתכונות

### אם נתקעת

1. בדוק את `OPEN_DECISIONS.md` - אולי החלטה נדרשת
2. בדוק את `JAPAN_TRIP.md` - אולי המידע שם
3. בדוק את `DESIGN_GUIDE.md` - אולי יש שם תשובה לעיצוב
4. שאל את Roy שאלה אחת ברורה ולא שלוש
5. אם זה החלטת UI, הצע 2-3 אופציות עם screenshots/wireframes
